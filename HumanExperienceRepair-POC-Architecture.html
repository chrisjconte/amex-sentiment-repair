Solution Overview

Design Principle: When an incident closes with negative employee sentiment, the platform automatically creates an Experience Repair Case and triggers an agentic AI workflow. Five specialist agents analyze the employee's history, explain the sentiment drivers, and compose a personalized outreach message — all before the SDM opens the record.

Detect (Flow Designer)

A Flow Designer flow monitors incident closures. When an incident closes with a negative sentiment indicator (Negative / Very Negative), the flow automatically creates an Experience Repair Case (ERC) linked to that incident and the affected colleague.

Analyze & Enrich (Agentic AI)

An agentic trigger fires on ERC creation. Five specialist agents run sequentially: Pattern Analyzer identifies failure patterns, Engagement Historian builds a 90-day timeline, Sentiment Explainer explains root causes, Repair Message Composer drafts the outreach, and ERC Case Summarizer writes the case summary.

Repair (SDM Action)

The SDM opens a fully enriched ERC — all AI analysis pre-populated. They review the AI-composed repair message, personalize if needed, and click "Send Repair Message" to initiate outreach. The case moves to Pending Outreach → Outreach Sent → Resolved.

2

Agentic Architecture

What makes this agentic? The workflow uses a ReActive Planner orchestration engine to dispatch five specialist agents sequentially. Each agent reads the current record state, decides which tools to call, writes results back to the ERC, and signals completion — then the orchestrator dispatches the next agent in the plan.

REACTIVE PLANNER ORCHESTRATION ENGINE
WHY

Business Problem

When employees have a poor IT experience, it often goes unaddressed — the SDM never knows. This workflow ensures every negative incident close automatically triggers a structured, AI-enriched repair process before the SDM opens the record.

HOW

Orchestration Engine

A ReActive Planner dispatches agents sequentially using a base_plan + objective_template injected at trigger time. Each agent reads state, calls tools, writes results, and signals done — the orchestrator advances to the next step.

WHO

Five Specialist Agents

Each agent owns exactly one area of analysis: pattern detection, engagement history, sentiment explanation, message composition, and case summarization. Agents run sequentially — each builds on the output of the prior.

WHAT

Output: A Pre-Enriched Case

By the time the SDM opens the ERC, all 7 AI fields are populated: pattern classification, sentiment drivers, engagement timeline, AI resolution plan, repair message, short description, and case description. State is Pending Outreach.

Known Issues & Resolutions

HITL Confirmation Dialog — Agent 4 (Repair Message Composer)

Symptom: Agent 4 pauses and asks "Is it okay to proceed with composing a message?" before writing u_repair_message. Requires human click to continue.

Root Cause: ReAct strategy enforces Human-In-The-Loop for any tool call it classifies as "composing communication to a person." This is platform-level behavior in the sn_aia_strategy engine — not controllable via agent instructions, role description, or tool description changes.

Resolution approach: Merge Agent 4's work into Agent 3's tool (single tool writes all remaining fields). Reduces to 3-agent plan. Eliminates the HITL step by removing the "composing communication" classification from any agent invocation. Under active testing.

Agent 5 (ERC Case Summarizer) Never Dispatched

Symptom: After Agents 1–4 complete, the orchestrator terminates the session without dispatching Agent 5. short_description and description remain empty.

Root Cause: ReActive Planner creates its execution plan at session start from base_plan + objective_template. Prior base_plan wording used "Enriched" as a terminal state label (matching the workflow name), causing the planner to treat Agent 4 completion as session end.

Resolution: Updated base_plan to explicit 3-step plan (or 5-step with unambiguous terminal condition: state=3). Removed "Enriched" language from step labels. objective_template updated with explicit case_sys_id in each step.

Agent 2 — case_sys_id Missing Error

Symptom: Agent 2 (Engagement Historian) failed with "build a 90-day engagement history failed because case_sys_id was missing" despite Agent 1 succeeding.

Root Cause: objective_template only explicitly stated case_sys_id = ${sys_id} in Step 3. Agent 2's tools require case_sys_id as mandatory input. The LLM could not reliably extract the sys_id from the header-level mention for Step 2.

Resolution: Added (case_sys_id = ${sys_id}) explicitly to Steps 1 and 2 in objective_template. Verified: Agent 2 now writes u_engagement_timeline successfully.

Trigger Deduplication — Trigger Doesn't Fire for Same Source Incident

Symptom: Creating a new ERC with the same source incident as an existing open ERC doesn't trigger the agentic workflow.

Root Cause: Platform-level deduplication in sn_aia_trigger_configuration prevents firing a new session when an active session for the same record already exists.

Workaround: Use a different source incident for each test ERC. Ensure prior test ERCs are closed or moved to a terminal state before re-testing.

Playwright Browser Conflict

Symptom: Playwright MCP cannot launch a controlled Chrome instance when Chrome is already running as a regular session. Error: "Opening in existing browser session."

Root Cause: Chrome must be fully quit (⌘Q, not just window closed) for Playwright to launch a controlled instance.

Workaround: Quit Chrome completely before running Playwright-based browser automation. Use Force Quit (⌘⌥⎋) if needed.
